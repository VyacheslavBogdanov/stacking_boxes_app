# Архитектура приложения Stacking Boxes Calculator

Документ описывает архитектуру клиентского приложения: структуру папок, слои, поток данных и конфигурацию.

## Стек технологий

| Область      | Технология                                |
| ------------ | ----------------------------------------- |
| UI-фреймворк | Vue 3 (Composition API, `<script setup>`) |
| Язык         | TypeScript 5 (strict mode)                |
| Сборщик      | Vite                                      |
| Стили        | SCSS (BEM)                                |
| Состояние    | Pinia                                     |
| HTTP-клиент  | Fetch / Axios (обёртка в `src/api/`)      |
| Бэкенд       | Express.js (Node 22)                      |
| Unit-тесты   | Vitest + @vue/test-utils                  |
| E2E-тесты    | Playwright                                |

## Структура папок

```
stacking_boxes_app/
├── e2e/                  — E2E-тесты Playwright (*.spec.ts)
├── public/               — статические ресурсы
├── src/
│   ├── api/              — HTTP-клиент и сервисы для работы с API
│   ├── components/       — Vue-компоненты
│   │   └── ui/           — базовые переиспользуемые компоненты (кнопки, инпуты)
│   ├── composables/      — переиспользуемая логика (use*)
│   ├── stores/           — Pinia-сторы (состояние приложения)
│   ├── styles/           — SCSS: переменные, миксины, глобальные стили
│   ├── types/            — TypeScript-типы и интерфейсы
│   ├── docs/             — документация проекта
│   ├── App.vue           — корневой компонент
│   └── main.ts           — точка входа приложения
├── vite.config.ts        — конфигурация Vite
├── vitest.config.ts      — конфигурация Vitest
├── playwright.config.ts  — конфигурация Playwright
├── tsconfig.json         — базовый конфиг TypeScript
├── tsconfig.app.json     — конфиг TypeScript для приложения
├── eslint.config.js      — ESLint 9 (flat config)
├── .prettierrc.json      — Prettier
├── .editorconfig         — настройки редактора
└── package.json          — зависимости и npm-скрипты
```

> Unit-тесты (`*.test.ts`) располагаются рядом с тестируемым файлом, а не в отдельной папке `__tests__/`.

## Слои приложения

Приложение разделено на слои с чёткой ответственностью. Каждый слой зависит только от нижележащих.

### Types — типы данных

Определения интерфейсов и типов, используемых во всех слоях: параметры коробки, марки картона, результаты расчёта, DTO запросов/ответов.

### API — взаимодействие с бэкендом

HTTP-клиент и сервисные функции для каждого эндпоинта. Инкапсулирует детали запросов (URL, метод, заголовки) и возвращает типизированные данные.

### Stores — управление состоянием (Pinia)

Централизованное хранение данных приложения: список марок картона, параметры формы, результаты расчёта, состояние авторизации администратора. Сторы вызывают API-сервисы и предоставляют реактивные данные компонентам.

### Composables — бизнес-логика для компонентов

Функции `use*`, которые комбинируют сторы с локальной логикой: валидация формы, управление состоянием загрузки, обработка ошибок. Composable возвращает реактивные данные и методы, готовые к использованию в шаблоне.

### UI Components — базовые компоненты (`components/ui/`)

Переиспользуемые компоненты без бизнес-логики: кнопки, инпуты, выпадающие списки, сообщения об ошибках. Принимают данные через props, уведомляют родителя через events.

### Feature Components — компоненты бизнес-логики (`components/`)

Компоненты, реализующие конкретные функции приложения: форма расчёта, панель результатов, админ-панель. Используют composables для получения данных и логики, UI-компоненты для отображения.

### App — корневой компонент

`App.vue` собирает feature-компоненты в единое приложение. `main.ts` создаёт экземпляр Vue, подключает Pinia и монтирует приложение.

## Поток данных

```
Бэкенд API
    ↕ HTTP (fetch)
src/api/        — сервисы API (запросы, маппинг DTO → типы)
    ↓
src/stores/     — Pinia-сторы (реактивное состояние)
    ↓
src/composables/ — composables (бизнес-логика, валидация)
    ↓
src/components/ — Vue-компоненты (отображение, пользовательский ввод)
    ↓
Template        — HTML-шаблон (рендеринг в DOM)
```

**Обратный поток (действия пользователя):**

```
Пользователь → Component (event) → Composable → Store → API → Бэкенд
```

## API-прокси

В режиме разработки Vite dev server проксирует все запросы к `/api` на бэкенд:

```
Клиент → /api/endpoint → Vite Proxy → http://81.94.156.176:5011/endpoint
```

Конфигурация прокси в `vite.config.ts`:

-   **Путь:** `/api`
-   **Целевой сервер:** `http://81.94.156.176:5011`
-   **Перезапись пути:** `/api` удаляется из URL (`/api/grades` → `/grades`)
-   **changeOrigin:** `true`

Это позволяет фронтенду обращаться к API через относительные пути (`/api/...`), избегая проблем с CORS при локальной разработке.

## Файлы конфигурации

| Файл                   | Назначение                                                  |
| ---------------------- | ----------------------------------------------------------- |
| `vite.config.ts`       | Сборка, dev-сервер, алиас `@/*`, API-прокси                 |
| `vitest.config.ts`     | Unit-тесты: окружение jsdom, наследует настройки Vite       |
| `playwright.config.ts` | E2E-тесты: базовый URL `localhost:5173`, headless-режим     |
| `tsconfig.json`        | Базовый конфиг TypeScript, ссылки на `app` и `node` конфиги |
| `tsconfig.app.json`    | TypeScript для src/: strict, DOM-типы, алиас путей          |
| `eslint.config.js`     | ESLint 9: Vue 3 essential + TypeScript + Prettier           |
| `.prettierrc.json`     | Форматирование: табы, 100 символов, одинарные кавычки       |
| `.editorconfig`        | Единые настройки редактора для всей команды                 |
| `nodemon.json`         | Конфигурация автоперезапуска Node.js                        |
